Documentation / Introduction

Introduction

Zinc provides financial delegation infrastructure that enables AI agents to handle payments autonomously while maintaining full control, audit trails, and policy enforcement.

📚 Complete Resource Guide
Access the full documentation index at docs.zinc.dev/llms.txt for a comprehensive list of all available resources and endpoints.

Overview

Zinc is built on the principle that AI agents need secure, programmable access to payment infrastructure without exposing sensitive credentials. Instead of sharing credit card numbers, agents declare their intent to make a purchase, receive a virtual card locked to that specific purpose, and complete the transaction within defined spending policies.

Key Concepts

Understanding these foundational concepts will help you integrate Zinc into your agent workflows:

Customer
The entity responsible for funding card spending. Customers set spending policies, add funding, and maintain oversight of all agent transactions.
Agent
An autonomous AI system that can request and utilize payment cards. Agents operate within customer-defined spending limits and merchant constraints.
Card
A virtual payment card provisioned for a specific intent. Cards can be single-use or recurring, with spending limits enforced at the network level.
Intent
A pre-transaction declaration of what an agent plans to purchase, enabling policy enforcement before financial details are exposed.
Policy
Customer-defined rules that govern agent spending behavior including per-transaction limits, daily caps, and merchant restrictions.

Documentation / Quickstart

Quickstart

Get started with Zinc and issue your first agent card in minutes.

1. Create Your Account

Sign up at zinc.dev and complete KYC verification. This typically takes 2-3 minutes with instant approval for most users.

# Using the Zinc CLI
zinc auth signup --email user@example.com

# Or via API
POST /v1/customers
{
  "type": "consumer",
  "email": "user@example.com",
  "name": {
    "first": "Alex",
    "last": "Rivera"
  }
}

2. Add Funding

Fund your account via ACH, wire transfer, or cryptocurrency. Deposits are typically available within 1-2 business days for ACH, instantly for crypto.

# Get your funding details
GET /v1/funding/details

# Response includes ACH routing, wire instructions, and crypto address

3. Connect Your Agent

Install the Zinc MCP Server or Skills package to give your agent payment capabilities:

# Via MCP (Claude Code, Cursor, etc.)
claude mcp add zinc --url https://mcp.zinc.dev/api/mcp

# Via Skills.sh
npx skills add zinc/zinc-mcp-skills

4. Issue Your First Card

Have your agent create an intent to receive a virtual card:

POST /v1/intents
{
  "purpose": "Purchase API credits for OpenAI",
  "expectedAmount": 5000,
  "expectedMerchant": "openai.com"
}

# Response - card issued automatically
{
  "intentId": "int_abc123",
  "status": "card_issued",
  "card": {
    "cardId": "card_xyz789",
    "last4": "4242"
  }
}

Documentation / Authentication

Authentication

Secure your API requests with bearer token authentication.

API Keys

All API requests require authentication using a bearer token. You can generate API keys from your dashboard at zinc.dev/dashboard/keys.

curl -X GET https://api.zinc.dev/v1/customers \
  -H "Authorization: Bearer sk_live_your_api_key" \
  -H "Content-Type: application/json"

Key Types

sk_live_*
Production keys for live transactions. Handle with care and never expose in client-side code.
sk_test_*
Test keys for development. Transactions use simulated funds and won't affect real accounts.

Security Best Practices

  • Store API keys in environment variables, never in source code
  • Rotate keys periodically and immediately if compromised
  • Use test keys during development
  • Implement IP allowlists for production keys

Documentation / Customers

Customers

A Customer represents the entity responsible for funding card spending. Customers can be either individuals or businesses.

Customer Types

consumer
An individual person using Zinc for personal spending delegation.
corporate
A business entity with multiple users and agents under a single account.

Creating a Customer

Create a new customer account with the required fields:

POST /v1/customers
{
  "type": "consumer",
  "email": "alex@example.com",
  "name": {
    "first": "Alex",
    "last": "Rivera"
  }
}

# Response
{
  "id": "cus_abc123",
  "type": "consumer",
  "email": "alex@example.com",
  "status": "pending",
  "createdAt": "2024-01-15T10:30:00Z"
}

Customer Fields

type
Either consumer for individuals or corporate for business entities.
email
Primary contact email address for the customer account.
name
Object containing first and last name for consumers, or business name for corporate.
status
KYC verification status of the customer account.

KYC Verification Status

Customers must complete KYC verification before issuing cards:

  • pending — Awaiting verification submission or review
  • verified — Approved for card issuance
  • rejected — Verification failed, may require additional documentation

Default Policies

Customers can establish default spending policies that cascade to all their agents and cards:

PATCH /v1/customers/cus_abc123
{
  "defaultPolicy": {
    "requireIntent": true,
    "autoApproveBelow": 5000,
    "limits": {
      "perDay": 100000
    }
  }
}

Documentation / Agents

Agents

An Agent is an autonomous actor—whether AI or automation—that makes purchases on behalf of customers. Agents require card assignments and must declare purchase intents before spending.

Creating an Agent

Register a new agent under a customer account:

POST /v1/agents
{
  "customerId": "cus_abc123",
  "externalId": "my-agent-001",
  "name": "Shopping Assistant"
}

# Response
{
  "id": "agent_xyz789",
  "customerId": "cus_abc123",
  "name": "Shopping Assistant",
  "status": "active",
  "createdAt": "2024-01-15T10:30:00Z"
}

Agent Fields

customerId
The ID of the customer account this agent belongs to.
externalId
Your internal identifier for this agent. Must be unique within the customer.
name
A human-readable display name for the agent.
status
Current operational status of the agent.

Agent Status

  • active — Full purchasing capabilities and card access
  • suspended — Temporary spending restrictions, no new transactions allowed

Policy Overrides

Agents can have custom policy settings that override customer-level defaults:

PATCH /v1/agents/agent_xyz789
{
  "policy": {
    "requireIntent": true,
    "limits": {
      "perAuth": 10000,
      "perDay": 50000
    },
    "blockedMccs": ["7995", "6211"]
  }
}

Suspending an Agent

Temporarily disable an agent's spending capabilities:

POST /v1/agents/agent_xyz789/suspend

# To reactivate
POST /v1/agents/agent_xyz789/activate

Documentation / Cards

Cards

Cards are virtual spending instruments issued to agents with configurable policy controls. They function as controlled spending tools within the Zinc system.

Card Types

single
One-time use cards that auto-close after a successful transaction. Ideal for one-off purchases.
multi
Reusable cards for recurring transactions like subscriptions or repeat purchases from the same merchant.

Creating a Card

POST /v1/cards
{
  "customerId": "cus_abc123",
  "agentId": "agent_xyz789",
  "usage": "single"
}

# Response
{
  "id": "card_def456",
  "customerId": "cus_abc123",
  "agentId": "agent_xyz789",
  "usage": "single",
  "status": "active",
  "last4": "4242"
}

Card Status

  • active — Card is usable for transactions
  • frozen — Temporarily blocked, can be unfrozen
  • closed — Permanently deactivated

Managing Card Status

# Freeze a card
POST /v1/cards/card_def456/freeze

# Unfreeze a card
POST /v1/cards/card_def456/unfreeze

# Permanently close a card
POST /v1/cards/card_def456/close

Retrieving Card Credentials

To access sensitive card details (PAN, CVV, expiry), the agent must first declare an intent:

# First, create an intent
POST /v1/cards/card_def456/intents
{
  "purpose": "Purchase software license",
  "expectedAmount": 9900,
  "expectedMerchant": "GitHub"
}

# Then retrieve credentials
GET /v1/cards/card_def456/credentials

# Response
{
  "pan": "4242424242424242",
  "cvv": "123",
  "expMonth": "12",
  "expYear": "2025"
}

Policy Controls

Cards support granular policy restrictions:

  • Spending limits — Maximum transaction amounts
  • Merchant whitelists — Restrict to specific vendors
  • Merchant locking — Lock card to first authorized merchant
  • MCC restrictions — Block specific merchant categories

Documentation / Intents

Intents

An Intent is a pre-transaction declaration of what an agent plans to purchase, enabling policy enforcement before financial details are exposed.

Purpose

Intents serve four key functions:

  • Approval workflows — Route significant purchases for human review
  • Merchant validation — Restrict transactions to approved vendors
  • Spending limits — Enforce caps proactively
  • Audit trails — Create transaction documentation with stated purposes

Creating an Intent

POST /v1/cards/card_def456/intents
{
  "purpose": "Purchase office supplies from Amazon",
  "expectedAmount": 5000,
  "expectedMerchant": "Amazon"
}

# Response
{
  "id": "int_abc123",
  "cardId": "card_def456",
  "purpose": "Purchase office supplies from Amazon",
  "expectedAmount": 5000,
  "expectedMerchant": "Amazon",
  "status": "pending"
}

Intent Fields

purpose
A descriptive rationale for the purchase. Used for audit trails and human review.
expectedAmount
The anticipated transaction amount in cents. System allows ±10% tolerance.
expectedMerchant
The intended vendor name. Uses fuzzy matching for name variations.

Intent Status Lifecycle

  • pending_approval — Awaiting human authorization
  • pending — Approved and awaiting transaction
  • matched — Transaction matched the declared intent
  • mismatched — Transaction deviated from intent (amount or merchant)
  • expired — Intent expired without a matching transaction

Managing Intents

# Approve a pending intent
POST /v1/intents/int_abc123/approve

# Reject a pending intent
POST /v1/intents/int_abc123/reject
{
  "reason": "Purchase not authorized"
}

# Cancel an intent
POST /v1/intents/int_abc123/cancel

Documentation / Policies

Policies

Policies establish governance rules for card utilization, controlling spending thresholds, merchant categories, approval procedures, and operational restrictions.

Policy Hierarchy

Policies cascade through three levels, with more specific settings overriding broader ones:

# Priority (highest to lowest)
Card Level → Agent Level → Customer Level

# Example: Card limit of $25 overrides Agent limit of $100

Intent & Approval Controls

requireIntent
When enabled, agents must declare a purpose before accessing card information.
requireApproval
Intents require human authorization before activation.
autoApproveBelow
Amount threshold (in cents) below which intents are auto-approved.
requireAttestation
Agents must confirm intent at each card access, generating timestamped records.

Spending Limits

Three limit types restrict expenditures (all values in cents):

perAuth
Maximum amount for a single transaction.
perDay
Maximum total spending in a 24-hour period.
perMonth
Maximum total spending in a calendar month.

Merchant Controls

allowedMerchants
Whitelist of merchant names. Only transactions at these vendors are processed.
lockToFirstMerchant
After initial purchase, restrict all subsequent transactions to that merchant.
allowedMccs
Whitelist of merchant category codes (e.g., 5411 for supermarkets).
blockedMccs
Blacklist of merchant category codes (e.g., 7995 for gambling).

Configuration Examples

High-Trust Agent

{
  "requireIntent": false,
  "limits": {
    "perAuth": 100000,
    "perDay": 500000
  }
}

Supervised Agent

{
  "requireIntent": true,
  "requireApproval": true,
  "autoApproveBelow": 2500,
  "limits": {
    "perAuth": 25000,
    "perDay": 100000
  },
  "cooldownMinutes": 15
}

Restricted Card

{
  "requireAttestation": true,
  "attestationWindowMinutes": 10,
  "allowedMccs": ["5812"],
  "limits": {
    "perAuth": 5000,
    "perDay": 10000
  },
  "lockToFirstMerchant": true
}

Documentation / KYC Verification

KYC Verification

Know Your Customer (KYC) verification is required before customers can issue cards or fund their accounts.

Verification Process

The KYC process varies based on customer type:

Consumer Verification

  • Government-issued ID (passport, driver's license)
  • Proof of address (utility bill, bank statement)
  • Identity verification selfie

Corporate Verification

  • Business registration documents
  • Proof of business address
  • Beneficial owner identification
  • Director/officer verification

Submitting KYC Application

POST /v1/customers/cus_abc123/kyc
{
  "idDocument": {
    "type": "passport",
    "frontImage": "base64_encoded_image..."
  },
  "addressDocument": {
    "type": "utility_bill",
    "image": "base64_encoded_image..."
  }
}

KYC Status

  • pending — Application submitted, under review
  • verified — Verification successful, full access granted
  • rejected — Verification failed, may resubmit with corrections

Transactions

Transactions represent completed or pending card authorizations with full audit details.

Listing Transactions

GET /v1/transactions?customerId=cus_abc123&limit=50

# Response
{
  "data": [
    {
      "id": "txn_abc123",
      "cardId": "card_def456",
      "amount": 4999,
      "merchant": "Amazon",
      "status": "settled",
      "intentId": "int_xyz789",
      "createdAt": "2024-01-15T10:30:00Z"
    }
  ],
  "hasMore": true
}

Transaction Fields

amount
Transaction amount in cents.
merchant
Merchant name as reported by the payment network.
mcc
Merchant Category Code identifying the business type.
intentId
The intent that was matched to this transaction (if any).
status
Current transaction status: pending, settled, declined, or refunded.

Documentation / Webhooks

Webhooks

Receive real-time notifications for events like card issuance, transaction authorization, and policy violations.

Configuring Webhooks

POST /v1/webhooks
{
  "url": "https://your-server.com/webhooks/zinc",
  "events": [
    "card.created",
    "transaction.authorized",
    "intent.approved",
    "intent.rejected"
  ]
}

Event Types

card.created
A new card has been issued.
card.frozen
A card has been frozen.
transaction.authorized
A transaction has been authorized.
transaction.declined
A transaction was declined due to policy or limits.
intent.created
A new intent has been created.
intent.approved
An intent has been approved.
intent.rejected
An intent has been rejected.

Webhook Payload

{
  "id": "evt_abc123",
  "type": "transaction.authorized",
  "createdAt": "2024-01-15T10:30:00Z",
  "data": {
    "transactionId": "txn_xyz789",
    "cardId": "card_def456",
    "amount": 4999,
    "merchant": "Amazon"
  }
}

Documentation / API Overview

API Overview

The Zinc API is organized around REST principles with JSON request and response bodies.

Base URL

https://api.zinc.dev/v1

Authentication

All requests require a bearer token in the Authorization header:

Authorization: Bearer sk_live_your_api_key

Request Format

POST /v1/resource
Content-Type: application/json

{
  "field": "value"
}

Response Format

Successful responses return the resource object:

{
  "id": "resource_abc123",
  "field": "value",
  "createdAt": "2024-01-15T10:30:00Z"
}

Pagination

List endpoints support cursor-based pagination:

GET /v1/transactions?limit=50&startingAfter=txn_abc123

Documentation / Error Handling

Error Handling

The API uses conventional HTTP response codes and returns detailed error messages.

HTTP Status Codes

200 OK
Request succeeded
201 Created
Resource created successfully
400 Bad Request
Invalid request parameters
401 Unauthorized
Missing or invalid API key
403 Forbidden
Insufficient permissions
404 Not Found
Resource doesn't exist
429 Too Many Requests
Rate limit exceeded
500 Internal Error
Server error (contact support)

Error Response Format

{
  "error": {
    "code": "invalid_request",
    "message": "The 'amount' field must be a positive integer",
    "param": "amount"
  }
}

Documentation / Code Examples

Code Examples

Complete implementation examples in popular languages.

Python

import requests

API_KEY = "sk_live_your_api_key"
BASE_URL = "https://api.zinc.dev/v1"

headers = {
    "Authorization": f"Bearer {API_KEY}",
    "Content-Type": "application/json"
}

# Create an intent and get card credentials
intent = requests.post(
    f"{BASE_URL}/cards/card_abc123/intents",
    headers=headers,
    json={
        "purpose": "Purchase API credits",
        "expectedAmount": 5000,
        "expectedMerchant": "OpenAI"
    }
).json()

credentials = requests.get(
    f"{BASE_URL}/cards/card_abc123/credentials",
    headers=headers
).json()

Node.js

const axios = require('axios');

const client = axios.create({
  baseURL: 'https://api.zinc.dev/v1',
  headers: {
    'Authorization': `Bearer ${process.env.ZINC_API_KEY}`
  }
});

// Create an intent
const { data: intent } = await client.post(
  '/cards/card_abc123/intents',
  {
    purpose: 'Purchase API credits',
    expectedAmount: 5000,
    expectedMerchant: 'OpenAI'
  }
);

// Get credentials
const { data: credentials } = await client.get(
  '/cards/card_abc123/credentials'
);
