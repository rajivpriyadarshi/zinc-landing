API Reference / Introduction

API Reference

The Zinc API is organized around REST. Our API has predictable resource-oriented URLs, accepts JSON-encoded request bodies, returns JSON-encoded responses, and uses standard HTTP response codes and authentication.

Base URL

All API requests should be made to the following base URL:

Base URL: https://api.zinc.dev/v1

Request Format

The API accepts JSON-encoded request bodies. Include the Content-Type: application/json header with all requests that include a body.

curl -X POST https://api.zinc.dev/v1/customers \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{
    "type": "consumer",
    "email": "user@example.com",
    "name": { "first": "Alex", "last": "Rivera" }
  }'

Response Format

All responses are returned as JSON with consistent structure including resource IDs, metadata, and ISO 8601 timestamps.

{
  "id": "cus_abc123",
  "type": "consumer",
  "email": "user@example.com",
  "status": "active",
  "createdAt": "2024-01-15T10:30:00Z"
}

Idempotency

Include an Idempotency-Key header on POST requests to prevent duplicate operations. Keys are cached for 24 hours.

curl -X POST https://api.zinc.dev/v1/cards \
  -H "Authorization: Bearer sk_live_..." \
  -H "Idempotency-Key: unique-request-id-123" \
  -H "Content-Type: application/json" \
  -d '{ ... }'
API Reference / Authentication

Authentication

The Zinc API uses bearer token authentication. All requests must include your API key in the Authorization header.

API Keys

You can generate API keys from your dashboard. There are two types of keys:

Key Type Prefix Description
Live sk_live_ Production keys for real transactions
Test sk_test_ Sandbox keys for development

Authorization Header

Include your API key in the Authorization header as a Bearer token:

Authorization: Bearer sk_live_your_api_key

Example Request

curl https://api.zinc.dev/v1/customers \
  -H "Authorization: Bearer sk_live_abc123xyz"

Authentication Errors

If authentication fails, you'll receive a 401 Unauthorized response:

{
  "error": {
    "code": "unauthorized",
    "message": "Invalid or missing API key"
  }
}
API Reference / Errors

Errors

The API uses conventional HTTP response codes to indicate success or failure of requests.

HTTP Status Codes

Code Status Description
200 OK Request succeeded
201 Created Resource created successfully
400 Bad Request Invalid request parameters
401 Unauthorized API key authentication failed
403 Forbidden Insufficient permissions
404 Not Found Resource doesn't exist
429 Too Many Requests Rate limit exceeded
500 Server Error Server-side issue

Error Response Format

{
  "error": {
    "code": "invalid_request",
    "message": "The 'amount' field must be a positive integer",
    "param": "amount"
  }
}
API Reference / Pagination

Pagination

List endpoints support cursor-based pagination for efficiently navigating large result sets.

Query Parameters

Parameter Type Description
limit integer Number of results per page (1-100, default: 50)
cursor string Pagination cursor from previous response

Response Fields

Field Type Description
data array Array of resource objects
nextCursor string | null Cursor for the next page, null if no more pages

Example

# First page
GET /v1/transactions?limit=50

# Next page using cursor
GET /v1/transactions?limit=50&cursor=eyJpZCI6InR4bl8xMjM0NSJ9
API Reference / Rate Limits

Rate Limits

The API enforces rate limits to ensure fair usage and maintain service stability.

Rate Limit Tiers

Plan Limit Window
Standard 100 requests per minute
Pro 1,000 requests per minute
Enterprise Custom negotiated

Rate Limit Headers

Responses include headers indicating your current rate limit status:

X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1704067200

Exceeding Limits

When you exceed the rate limit, you'll receive a 429 Too Many Requests response:

{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "Too many requests. Please retry after 60 seconds."
  }
}

API Reference / Customers / Create

Create Customer

Create a new customer account.

POST /v1/customers

Creates a new customer and returns the customer object.

Request Body

Parameter Type Description
typerequired string Customer type: consumer or corporate
emailrequired string Customer email address
namerequired object Name object with first and last fields
phone string Phone number in E.164 format
metadata object Custom key-value pairs

Example Request

curl -X POST https://api.zinc.dev/v1/customers \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{
    "type": "consumer",
    "email": "alex@example.com",
    "name": {
      "first": "Alex",
      "last": "Rivera"
    },
    "phone": "+14155551234"
  }'

Response

201 Returns the created customer object

{
  "id": "cus_abc123",
  "type": "consumer",
  "status": "pending",
  "email": "alex@example.com",
  "name": {
    "first": "Alex",
    "last": "Rivera"
  },
  "phone": "+14155551234",

List Webhooks

List all configured webhook endpoints.

GET /v1/webhooks

Create Webhook

Register a new webhook endpoint.

POST /v1/webhooks

Request Body

Parameter Type Description
urlrequired string HTTPS endpoint URL
eventsrequired array Event types to subscribe to

Available Events

  • card.created - Card issued
  • card.frozen - Card frozen
  • card.closed - Card closed
  • transaction.authorized - Transaction authorized
  • transaction.declined - Transaction declined
  • intent.created - Intent created
  • intent.approved - Intent approved
  • intent.rejected - Intent rejected

Delete Webhook

Remove a webhook endpoint.

DELETE /v1/webhooks/{webhookId}
